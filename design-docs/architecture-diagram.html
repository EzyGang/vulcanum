Vulcanum — Agent Orchestrator Architecture

Task manager → Orchestrator → Worker → Sandbox → Harness

How work flows

  • 1. Task created in Kaneo/Linear with vulcanum:ready label
  • 2. Main App polls task manager, enriches with harness config
  • 3. Worker polls GET /poll every 15s, receives work
  • 4. Harness reads secrets from agent-vault proxy (localhost:8200)
  • 5. Result posted back, status synced to task manager

Security model

  • • Secrets live in Infisical, never touch Vulcanum Main App
  • • agent-vault proxy on worker handles auth, caching, rotation
  • • Harness reads localhost:8200 — no secrets over the wire
  • • Sandbox: no network by default, localhost:8200 only for secrets
  • • Output sanitized for secret patterns before submission

Why polling, not WebSocket

  • • Stateless HTTP — any load balancer can handle any request
  • • No connection state, no reconnection storms on restart
  • • Configurable interval (15s default, tune per workload)
  • • Works through every proxy and firewall
  • • SSE available as optional upgrade for near-real-time

Agent-native control

  • • vulcanum-api SKILL.md shipped in CLI binary
  • • 1:1 API mapping — every endpoint callable from agent
  • • vulcanum install-skill --agent hermes
  • • "Create a vulcanum task to fix clippy warnings"
  • • "Why did work item abc123 fail?"
